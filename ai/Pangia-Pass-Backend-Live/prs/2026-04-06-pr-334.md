## 🚀 Core Features

### Soft Rate Limits (Monitoring Only)
- **100 SMS/day per user** - Log warnings but allow requests (preserve UX)
- **50 voice calls/day per user** - Monitor usage without blocking
- Firebase transactions prevent race conditions (ADR-021)
- Structured logging for analytics and alerting

### Hard Rate Limits (Enforced)
- **3 number provisioning/day** - Prevent bulk abuse
- HTTP 429 responses with Retry-After header
- Atomic check-and-increment using Firebase transactions

### Provider API Rate Limiting
- **CloudNumbering: 50 req/s** - Token bucket rate limiter
- Request throttling via Cloud Tasks queues (Story 1.10)
- Exponential backoff + jitter for retry logic
- Prevents 429 errors from provider APIs

### Abuse Detection System
- **Spike Detection:** >50 SMS/hour flags potential spam
- **Velocity Detection:** >200 SMS/day (2x soft limit) triggers alert
- **Geographic Anomalies:** SMS to 10+ countries in 24h flagged
- Alert severity levels: low/medium/high/critical
- Manual review required before action (no auto-blocks)

### Admin Dashboard API
- `GET /api/admin/abuse/flagged` - List users with abuse alerts (pagination + filtering)
- `GET /api/admin/abuse/alerts/{alertId}` - Get detailed alert information
- `GET /api/admin/users/{userId}/usage` - User rate limit statistics
- Admin authentication via X-Admin-Token header

## 🏗️ Architecture Decisions

### Firebase RTDB Schema (ADR-021 Compliant)
```javascript
/telephony/
├── rateLimit/
│   ├── daily/{userId}/{date}/          // Daily counters
│   │   ├── smsCount: number
│   │   ├── callCount: number
│   │   ├── numberProvisioningCount: number
│   │   └── updated_at: ISO 8601
│   └── hourly/{userId}/{hour}/         // Hourly for spike detection
│       ├── smsCount: number
│       └── updated_at: ISO 8601
│
└── abuseDetection/
    └── alerts/{alertId}/               // Abuse alerts
        ├── userId: string
        ├── type: "spike" | "velocity" | "geographic"
        ├── severity: "low" | "medium" | "high" | "critical"
        ├── timestamp: ISO 8601
        ├── details: object
        └── resolved: boolean
```

### Transaction Safety Pattern
All counter increments use Firebase transactions to prevent race conditions when multiple SMS/calls happen concurrently:
```python
async def increment_daily_sms_count(user_id: str, date: str) -> int:
    def transaction_fn(current_data):
        if current_data is None:
            return {"smsCount": 1, "updated_at": now()}
        current_data["smsCount"] = current_data.get("smsCount", 0) + 1
        current_data["updated_at"] = now()
        return current_data
    
    return await firebase.update_with_transaction(path, transaction_fn)
```

## 🎓 Philosophy: "Soft Limits, Monitoring-First"

Pangia Pass is committed to providing an "unlimited" telephony experience that doesn't nickel-and-dime users with arbitrary restrictions. Our rate limiting strategy is built on **monitoring and observation** rather than aggressive blocking.

**Core Principles:**
1. **User Experience First** - Never block legitimate users for hitting soft limits
2. **Observe Before Acting** - Log patterns, analyze trends, investigate before action
3. **Manual Review Required** - No automatic suspensions without human verification
4. **Transparent Limits** - Hard limits clearly communicated to users
5. **Fair Use, Not Abuse** - Target actual abuse patterns, not power users
