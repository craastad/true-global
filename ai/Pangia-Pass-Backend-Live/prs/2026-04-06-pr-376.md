## Story 3.2 — Telnyx Full Provider Adapter (SMS + Numbers + Voice Delegation)

### Goal
We needed to promote Telnyx from partial/isolated usage to a first-class telephony provider in the shared provider architecture, so the platform can support multi-provider behavior (search, provisioning, SMS, webhooks, health) with consistent models and security.

### System context
The telephony backend is built around a provider adapter pattern:

Client/API -> Provider Factory -> Provider Adapter -> External Provider API -> Firebase/FCM

This story adds a full Telnyx adapter so Telnyx behaves like other providers in the same contract, instead of custom one-off logic.

### Functional behavior implemented

- **Telnyx API client**  
  Added async API v2 client with bearer auth, retry/backoff, and normalized provider error handling.

- **Numbers support**  
  Implemented number search, order/provision, and release handling in `TelnyxAdapter`.  
  Search results are normalized into platform `PhoneNumber` models.

- **SMS support**  
  Implemented outbound send and inbound/status webhook parsing in `TelnyxAdapter`, normalized into platform `SMSMessage` and status payloads.

- **Message ID consistency**  
  Delivery and outbound IDs are normalized with `telnyx_` prefix to keep provider-scoped message IDs consistent in Firebase updates.

- **Voice support via delegation**  
  `initiate_call`, `end_call`, `get_call_status`, `send_dtmf` delegate to existing `TelnyxCallControlService` (no duplicate voice stack).

- **Webhook security**  
  Added `TelnyxPublicKeyVerifier` (Ed25519 + timestamp checks), registered in webhook verifier registry for generic routes:
  - `/api/v1/telephony/webhooks/telnyx/sms/inbound`
  - `/api/v1/telephony/webhooks/telnyx/sms/status`

- **Provider registration**  
  Added Telnyx to `PROVIDER_CONFIGS` with required/optional secrets and capabilities metadata.

### Runtime flow (high level)

- **Outbound SMS**  
  API/service selects provider -> `TelnyxAdapter.send_sms()` -> Telnyx `/v2/messages` -> normalized SMS model persisted/tracked.

- **Inbound SMS webhook**  
  Telnyx -> generic webhook endpoint -> Telnyx signature verifier -> `TelnyxAdapter.handle_inbound_sms_webhook()` -> normalized message -> existing storage/notification path.

- **Delivery status webhook**  
  Telnyx -> generic status endpoint -> signature verifier -> `TelnyxAdapter.handle_delivery_status_webhook()` -> normalized status update in Firebase.

- **Number lifecycle**  
  Search -> `search_numbers()`; provision -> `create_number_order()` + Firebase save; release -> cooldown-safe release path (provider delete gated by config flag).

### Secrets/config required
- Required:
  - `TELNYX_API_KEY_DEV/PROD`

  - `TELNYX_VOICE_CONNECTION_ID_DEV/PROD` (or fallback `TELNYX_SIP_CONNECTION_ID_*`)

### Scope boundaries / non-goals in this PR
- This PR focuses on adding Telnyx as a full adapter and webhook/security integration.
- It does **not** redesign all API endpoints to dynamically choose Telnyx for current default CloudNumbering-specific paths.

### Risks and mitigations
- **Risk:** webhook signature misconfiguration in env.  
  **Mitigation:** verifier has explicit errors in prod when key missing/invalid; unit tests cover valid/invalid/expired signatures.
- **Risk:** number release behavior expectations differ (cooldown vs hard delete).  
  **Mitigation:** hard delete is behind explicit config flag; default is safe cooldown policy.
- **Risk:** environment-secret mismatch (DEV/PROD suffix).  
  **Mitigation:** SecretManager uses env-aware naming; validated with local + live integration tests.

### Rollout checklist
- Add/update required DEV/PROD secrets in GCP.
- Ensure Telnyx Messaging Profile webhook points to:
  - `.../api/v1/telephony/webhooks/telnyx/sms/inbound`
- Optionally configure outbound/status event routing to:
  - `.../api/v1/telephony/webhooks/telnyx/sms/status`
- Deploy to dev and run smoke checks:
  - search numbers
  - send SMS
  - webhook delivery status update
  - health check

### Rollback plan
If issues are found, disable Telnyx by removing/disabling Telnyx entry in `PROVIDER_CONFIGS` and redeploy; existing providers remain unaffected.
